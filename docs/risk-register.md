# Реестр рисков, багов и несостыковок

## Назначение

Этот документ фиксирует список основных рисков, багов, несостыковок и обязательных улучшений, выявленных при аудите EventDesign.

Он нужен не как историческая справка, а как рабочая матрица приоритетов.

## Классы приоритета

- **Blocker** — мешает запуску, демо или ломает ключевой end-to-end flow.
- **High** — архитектурно или security-критично, сильно снижает надёжность.
- **Medium** — заметно портит качество, но не полностью блокирует показ.
- **Low** — желательно улучшить, но не это решает судьбу проекта.

## Blocker

### B1. Race-condition при старте docker compose

Симптомы:

- сервисы иногда стартуют, иногда падают;
- приложение поднимается со второй или третьей попытки;
- сервисы стартуют раньше готовности Postgres/Redis/NATS/MinIO.

Причина:

- отсутствуют healthchecks;
- `depends_on` используется без `service_healthy`;
- нет детерминированной последовательности старта.

Что нужно сделать:

- healthchecks;
- `depends_on` long syntax;
- `condition: service_healthy`.

### B2. Гонки миграций и несогласованность схемы

Симптомы:

- сервисы падают с SQL-ошибками;
- часть таблиц существует, часть нет;
- старт зависит от порядка запуска сервисов.

Причина:

- миграции применяются не через отдельный migrator;
- несколько сервисов могут пытаться менять схему;
- порядок миграций или формат имён может быть некорректным.

Что нужно сделать:

- one-shot `db-migrator`;
- единый путь применения миграций;
- проверка SQLx migration files.

### B3. Нестабильный outbox -> JetStream -> projector pipeline

Симптомы:

- события не доходят до projections;
- read-side экраны не обновляются;
- редоставка создаёт дубли.

Причина:

- не создан stream/consumer;
- consumer ack’ает слишком рано;
- отсутствует идемпотентность projector’ов.

Что нужно сделать:

- bootstrap JetStream;
- `processed_messages`;
- ack after commit;
- idempotent projection updates.

### B4. Export pipeline доходит до completed, но файл не скачать

Симптомы:

- export job “вроде завершён”;
- файл не существует;
- presigned URL невалиден или ведёт в пустоту.

Причина:

- bucket не создан;
- MinIO настроен неверно;
- job status не соответствует реальному состоянию storage;
- download flow не проверяет или не находит объект.

Что нужно сделать:

- bootstrap bucket;
- строгая связка job status <-> storage object;
- корректный presigned URL flow.

### B5. Cookie auth + CSRF + CORS нестабильны

Симптомы:

- login/register/logout ведут себя нестабильно;
- браузер шлёт 401/403 без понятной причины;
- запросы с `credentials: include` ломаются.

Причина:

- неверные cookie flags;
- слишком permissive или несовместимый CORS;
- CSRF flow реализован неполно или неконсистентно.

Что нужно сделать:

- нормализовать cookie flags;
- задать strict CORS allowlist;
- довести CSRF flow до стабильного состояния.

## High

### H1. Протечки сервисных границ

Симптомы:

- Edge API ходит в БД напрямую;
- часть доменной логики не изолирована в нужных сервисах.

Риск:

- архитектура “на словах” одна, а в коде другая;
- сложно отлаживать и масштабировать.

Что нужно сделать:

- вынести direct DB access из Edge API;
- оставить Edge как BFF/orchestration layer.

### H2. Нет сквозной request/trace корреляции

Симптомы:

- запрос сложно проследить между Edge, gRPC-сервисами и worker’ами;
- логи есть, а связывать их неудобно.

Риск:

- плохая диагностируемость;
- observability выглядит слабой.

Что нужно сделать:

- request_id на Edge;
- gRPC metadata propagation;
- structured logs с корреляцией.

### H3. Ownership/BOLA риски

Симптомы:

- endpoints по id могут пропускать чужие объекты;
- DTO могут отдавать лишние внутренние поля.

Риск:

- утечка данных между пользователями;
- архитектурно слабая безопасность.

Что нужно сделать:

- проверить все object-level endpoints;
- ввести строгие ownership checks;
- зачистить DTO.

## Medium

### M1. Stale cache / stale dashboard / stale calendar

Симптомы:

- read-side отображает старые данные;
- dashboard и calendar отстают дольше ожидаемого.

Причина:

- отсутствует cache invalidation;
- нет TTL;
- связь между projection update и cache refresh не доведена.

Что нужно сделать:

- TTL или event-driven invalidation;
- пересмотреть горячие кэши.

### M2. Нет единого error envelope

Симптомы:

- фронтенд получает ошибки в разных форматах;
- тяжело обрабатывать и показывать их одинаково.

Что нужно сделать:

- ввести единый JSON-формат ошибок;
- включить `code`, `message`, `request_id`.

### M3. Неполные README и docs

Симптомы:

- новому человеку непонятно, как поднять проект;
- защита зависит от устных пояснений.

Что нужно сделать:

- обновить `README.md`;
- обновить `frontend/README.md`;
- обновить `docs/*`;
- перевести всё на русский язык.

## Low

### L1. Неполный observability baseline

Симптомы:

- базовые метрики есть, но не хватает удобной полной картины;
- нет log aggregation или OTel-детализации.

Что нужно сделать:

- довести request correlation;
- при возможности улучшить trace story;
- Loki можно оставить опциональным.

### L2. Недостаточно smoke/integration tests

Симптомы:

- система кажется рабочей, но regression легко проскальзывает;
- нет повторяемого доказательства работоспособности.

Что нужно сделать:

- добавить `scripts/smoke.sh`;
- добавить backend/frontend smoke/integration tests.

---

## Обязательные нововведения

Следующие улучшения должны считаться не “приятным бонусом”, а частью целевого состояния:

1. Детальный `db-migrator` и детерминированный startup.
2. Формализованная идемпотентность projector/export consumers.
3. Request id propagation как архитектурный стандарт.
4. Безопасный export pipeline через MinIO + presigned URL.
5. Smoke script для полного happy-path.
6. Перевод всех docs и README на русский язык.
7. Актуальный runbook и demo-script.

## Порядок исправлений

Чинить нужно строго в таком порядке:

1. startup и migrations;
2. event backbone и projections;
3. auth/security/export;
4. service boundaries, observability, docs;
5. tests, smoke, polish.

Если перепутать порядок, проект можно красиво сломать, но это и без нас у людей обычно неплохо получается.
